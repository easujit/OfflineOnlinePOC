Offline Hospitals System Design

Build a hospital web app that keeps working even when the internet is gone for 2–3 days. Doctors and staff should still be able to open the app, see patients, add notes/reports, print, and continue workflows. When the internet returns, everything syncs safely—no data loss, no duplicates.

Big Picture (Tiered, Offline‑First)

We use a 3‑tier design so we are safe at every level:

  1. Device (PWA in the browser) – Works offline by itself
  • Service Worker (SW) caches the app shell → app always opens.
  • IndexedDB stores structured data (patients, vitals, meds, etc.).
  • OPFS (Origin Private File System) stores large files (PDFs, images, DICOM slices) locally.
  • "Outbox" pattern: every change you make is queued locally for later upload.

  1. Hospital Edge Node (small local server on LAN) – Optional but strongly recommended for 2–3 day outages
  • A tiny, low‑cost box (Intel NUC / Mini‑PC / Raspberry Pi 4) running Docker.
  • Hosts a Local Sync Service and a lightweight DB (e.g., SQLite/CR‑SQLite or CouchDB).
  • Acts like a nearby cloud: browsers connect to this when the real internet is down.
  • Stores big blobs (reports, images) locally to avoid filling staff devices.
  • When the internet is back, it syncs to the cloud.

  1. Cloud (multi‑tenant, cost‑efficient) – Single shared platform (no separate DB per hospital)
  • API + Sync Orchestrator
  • PostgreSQL (row‑level security with tenant_id) for structured data.
  • Object Storage (S3/Wasabi/B2) for large files.
  • Message Bus (optional; Kafka/NATS) for downstream jobs.

This gives us three safety nets: Device → Edge → Cloud.

Why this solves the 4 challenges

1) 2–3 days offline:

  • PWA + Edge Node keeps the system running locally for days.
  • Staff keep working on the LAN; data piles up locally (not lost).

2) Huge data volumes:

  • Use OPFS on the device for small/medium files, and Edge Node for large files.
  • Upload in chunks later; throttle and resume automatically.

3) No per‑hospital dedicated cloud DB:

  • Use one multi‑tenant cloud database (PostgreSQL) with tenant_id and row‑level security.
  • Each hospital has a logical partition, not a separate DB. Cheaper & simpler.

4) Patient safety:

  • Apps always open, local reads/writes always work.
  • No write is ever dropped (Outbox).
  • Deterministic conflict rules + review queue → no silent overwrites.

Data Placement (what lives where?)

  • Browser (PWA): most recent patients, today’s schedules, active in‑progress forms, small attachments, thumbnails.
  • Edge Node: full hospital slice for the last few weeks/months + all large recent files (lab PDFs, images, DICOM).
  • Cloud: full system of record for all hospitals; long‑term storage; analytics.

You can tune how much each tier keeps (size or days). Example:

  • Device: 7 days or 2–5 GB per user.
  • Edge Node: 30–90 days or 200–500 GB per hospital.
  • Cloud: everything.

Offline‑First App Mechanics (step‑by‑step)

A) App Load (even when offline)

  1. Staff opens the URL or installed PWA.
  2. Service Worker serves cached HTML/JS/CSS → UI opens instantly.
  3. App reads from IndexedDB to show last known data.

B) Read Data

  1. Show cached lists (patients, visits, labs) from IndexedDB.
  2. If online → pull updates from Edge/Cloud → merge into IndexedDB.

C) Write Data (Outbox pattern)

  1. User action (add note, vitals, order, upload file).
  2. App writes to IndexedDB immediately → UI updates instantly.
  3. The app also writes a Mutation to Outbox {uuid, entity, patch, base_version, actor, timestamps}.
  4. If large file: save to OPFS; generate a file manifest (size, hash, chunks).

D) Sync Loop (transparent to user)

  1. Push up: send Outbox mutations to Edge/Cloud with Idempotency‑Key = uuid.
  2. Pull down: fetch remote changes since last_cursor.
  3. Apply merges; if conflict → create a conflict task (see below).
  4. For big files, use chunked uploads (resume if interrupted).

E) Recovery after 3‑day outage

  • The device first syncs with Edge Node; Edge syncs with Cloud.
  • If no Edge Node, device syncs directly to Cloud when WAN is back.
  • Idempotent mutations guarantee no duplicates; version checks prevent silent overwrites.

Local Storage Strategy (handle lots of data safely)

Structured data → IndexedDB (via an ergonomic wrapper like Dexie/RxDB).
Large files → OPFS (browser‑private file system) or Edge Node when available.
Cache size control → use LRU by data type.
Space estimation → use navigator.storage.estimate(); ask for Persistent Storage (navigator.storage.persist()), so the browser is less likely to evict data.
Compression → compress JSON payloads and thumbnails.
Pruning → keep only what’s active/recent on the device; keep more on Edge Node.

Tip: On iOS/Safari, installing the app as a PWA and requesting persistent storage improves reliability. For very large imaging workloads, prefer the Edge Node.

Sync Protocol (simple, robust, idempotent)

Objects: Patients, Encounters, Orders, Labs, Meds, Notes, Attachments, Users…
Each object has: id (UUIDv4), tenant_id, version, updated_at, updated_by, tombstone (for deletes).

Push (client → server)

  • Send Mutation: {idempotency_key, entity, entity_id, base_version, patch, actor, timestamps}.
  • Server checks idempotency_key → dedupe.
  • If base_version matches server version → apply patch → bump version.
  • If not → conflict (see below).
  • Server returns {commit_version, cursor}.

Pull (server → client)

  • Client requests /changes?since=<cursor>&tenant_id=<X>&datasets=<...>
  • Server returns changes (ordered) + new cursor.

Big files

  • Use chunked upload (tus.io‑style or S3 multipart).
  • Store file manifest {file_id, size, sha256, parts[]} and resume on failure.

Ordering

  • Prefer push‑then‑pull on resume, so user’s actions get committed quickly, then we catch up with others’ changes.

Conflict Resolution (no loss, no duplicates) : Conflicts happen when two users edit the same record offline.

Principles

  1. Never delete user input. Keep both versions if unsure.
  2. Make common cases automatic, rare cases go to a review queue.

Per‑entity rules (examples)

  • Vitals, Lab Results, New Notes → append‑only (no conflict). Use idempotency_key to prevent duplicates.
  • Patient Demographics → field‑level merge with last‑writer‑wins per field + role priority (Doctor > Nurse for specific fields), else flag for review.
  • Medications/Orders → treat as versioned items; if base_version mismatch:
  • If edits touch different fields → merge.
  • If the same field → mark conflict with a side‑by‑side diff for a supervisor.
  • Deletions → use tombstones; if someone edited a deleted record → keep both (as “restored” vs “deleted”) and ask the reviewer to decide.

Review UI

  • Show who changed what and when (actor, timestamps, device).
  • Offer Accept A / Accept B / Merge fields options.
  • Keep an audit trail of decisions.

Security & Privacy (simple checklist)

  • All traffic TLS (LAN & WAN).
  • At rest encryption:
  • Device: encrypt IndexedDB/OPFS with WebCrypto (AES‑GCM) using a key derived from user credentials + device secret.
  • Edge Node disks encrypted (LUKS/BitLocker).
  • Cloud DB and object storage encrypted.
  • Least privilege: row‑level security by tenant_id; RBAC for roles (Doctor, Nurse, Lab, Admin).
  • Offline auth: short‑lived cached tokens + device binding; set policies (e.g., 48–72h offline access).
  • Break‑glass account for emergencies (with heavy audit logging).
  • Audit log for all reads/writes, including offline writes committed later.
  • PII/PHI minimization on device; allow quick remote wipe when device is lost.

Cost‑Effective Choices

  • One multi‑tenant cloud (no per‑hospital DBs). Use PostgreSQL partitions by tenant_id.
  • Object storage like Backblaze B2 / Wasabi (cheaper than S3) for files.
  • Small Edge Node (8–16 GB RAM, 512 GB–2 TB SSD). One per hospital is usually enough.
  • Open‑source first: PostgreSQL, Nginx, MinIO (optional), Keycloak (optional), Dexie/RxDB on client.
  • Start with 2–3 modest VMs instead of Kubernetes. Add k8s only when scale demands.

Reference Implementation (technology sketch)

Frontend (PWA)

  • React/Vue + Service Worker (Workbox)
  • IndexedDB wrapper: Dexie or RxDB (PouchDB adapter)
  • File storage: OPFS + thumbnails in IndexedDB
  • Background sync: Workbox Background Sync / periodic sync (polyfill with timers)

Edge Node

  • Docker Compose services:
  • edge-api (FastAPI/Node) + edge-sync
  • sqlite or couchdb
  • minio (optional, for local files)
  • nginx (reverse proxy, TLS)

Cloud

  • API (Django/FastAPI) with idempotent mutations
  • PostgreSQL (RLS by tenant_id, partitions)
  • Object store (B2/Wasabi/S3) with multipart uploads
  • Optional: Kafka/NATS for async processing

Example Flows

1) Recording Vitals Offline

  1. Nurse enters vitals → saved to IndexedDB; outbox gets a mutation.
  2. Internet down: UI shows a small “Pending” badge.
  3. Later, Edge/Cloud reachable → push mutation with idempotency key.
  4. Server commits; returns new version; device updates local record and clears pending.

2) Uploading a Lab PDF (Large File)

  1. File saved to OPFS; a manifest is stored (sha256, size).
  2. Sync uses chunked upload (e.g., 5–10 MB parts).
  3. If network drops, resume from the last confirmed part.
  4. When complete, server verifies checksum; marks file as available.

3) Two People Edit Same Demographics

  1. Doctor A (offline) changes phone number; Nurse B (online) changes address.
  2. When A syncs later: base_version mismatch only for phone → merge fields; no conflict.
  3. If both changed the same field (phone), create a conflict task. Supervisor resolves.

Operational Playbook

  • Daily: automatic local backups on Edge Node; verify free disk space, rotate logs.
  • Weekly: disaster drill—simulate WAN outage; verify full function on LAN.
  • Monitoring: simple dashboards on Edge & Cloud; alert if sync stalled > N hours.
  • Onboarding a new hospital: add tenant_id, allowlist LAN, ship a pre‑imaged Edge Node (optional).

What if we don’t deploy an Edge Node?

It still works for many sites with lighter imaging needs:

  • PWA + IndexedDB + OPFS can handle forms, notes, vitals, prescriptions for days.
  • For heavy imaging (radiology), storage on staff tablets/laptops may be insufficient. In that case, Edge Node is strongly recommended.

Variations / Alternatives (with trade‑offs)

  1. PouchDB (Browser) ⇄ CouchDB (Edge/Cloud)
  • Easiest bi‑directional sync, built‑in conflict docs.
  • Great for JSON docs; less ideal for strict relational joins.
  1. CR‑SQLite / ElectricSQL (Browser WASM) ⇄ Postgres
  • True relational + CRDT replication; modern, efficient.
  • Slightly higher setup complexity; powerful conflict‑free merges.
  1. Custom Outbox + REST/GraphQL (as described)
  • Maximum control and fits existing stacks (Django/FastAPI).
  • You must implement conflicts, cursors, chunked uploads yourself.

Pick based on team skills and time. For fastest results, #1 or #3 are usually best.

Step‑by‑Step Rollout Plan

  1. MVP (2–4 weeks): PWA shell, IndexedDB cache, Outbox for notes/vitals, simple push/pull, idempotency, basic conflict UI, OPFS for PDFs, chunked upload.
  2. Edge Node (4–6 weeks): Dockerized Edge API + DB, LAN discovery, failover routing, background reconciliation to cloud.
  3. Hardening (continuous): encryption keys, RLS, audit logs, backup/restore drills, monitoring/alerts.
  4. Scale & Optimize: partitions by tenant, file lifecycle rules (auto‑archive old files), add background workers for OCR/Lab parsing.

Final Thought

With this design, the hospital keeps running even during long outages. Staff don’t change how they work; the system quietly stores and later syncs. Patients stay safe, and the organization avoids expensive per‑hospital infrastructure while still scaling to many sites.
